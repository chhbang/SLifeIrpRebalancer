## What the app does

`SLifeIrpRebalancer` is a Windows desktop app that produces AI-generated rebalancing proposals for a Samsung Life Insurance IRP (개인형 퇴직연금 / Korean individual retirement pension) account. The pipeline runs end-to-end inside the app:

1. **HTML → CSV.** User uploads the Samsung Life product-list HTML; the app parses it into two CSVs: principal-guaranteed products (`원리금보장형`) and funds (`펀드`). The fund CSV must preserve return-period columns dynamically (e.g. `수익률(3개월)`, `수익률(3년)`) because the AI uses these to reason about portfolios.
2. **Account input.** User enters total balance + per-holding amounts, with the CSV from step 1 driving autocomplete.
3. **Sell-decision UI.** Per holding, user picks 전량 매도 / 부분 매도 / 유지. A separate radio choice declares the rebalancing as either *immediate* or *maturity-reservation* (만기 예약) — this materially changes the prompt sent to the AI.
4. **AI proposal.** App builds a structured prompt (system instructions + product universe + holdings + sell decisions + user free-text) and queries Gemini, GPT, or Claude via the user's API key.
5. **PDF export** of the resulting proposal.

## Architecture (planned)

- **Platform:** WinUI 3 Blank App (Packaged), .NET 10 (or .NET 9), C# + XAML, Visual Studio 2026 on Windows 11.
- **Pattern:** MVVM via `CommunityToolkit.Mvvm`. The 5 screens are hosted in a single `NavigationView` shell.
- **Libraries:**
  - `HtmlAgilityPack` for parsing the Samsung Life product HTML.
  - `QuestPDF` (Fluent API) for the proposal PDF.
  - `HttpClient` + `System.Text.Json` for AI REST calls — no provider SDKs.
- **Five screens:** Settings (AI provider + API key, lifelong-annuity flag) → Data Preparation (HTML→CSV) → My Account → Sell Targets & Timing → AI Rebalance.

### Two cross-cutting constraints to preserve

These are the parts of the spec most likely to be lost or watered down during implementation — keep them intact:

- **Manager-name (운용사) extraction.** Product names are prefixed with the asset manager (`삼성`, `미래에셋`, etc.). Extract this into a separate column via regex / mapping table. It is *load-bearing*: if the user checks "lifelong annuity payout" in Settings, the prompt must constrain the AI to recommend only products where 운용사 is `삼성생명`.
- **Sell decisions as hard constraints in the prompt.** The prompt must explicitly tell the AI that the cash available for new purchases equals the sum of the user's sell decisions (full sell + partial sell amounts) — the AI is not free to override the user's keep/sell choices. The "immediate vs. maturity-reservation" radio also flows into the prompt as a framing instruction.

### Data models (from the spec)

- `AccountStatusModel` — `TotalAmount` (required), `DepositAmount?`, `ProfitAmount?`, `RebalanceTiming` (enum: `Immediate` | `MaturityReservation`), `OwnedItems: List<OwnedProductModel>`.
- `OwnedProductModel` — `ProductName`, `CurrentValue` (required), `ReturnRate?`, `AnnualizedReturn?`, `InvestedDays?`, `TotalShares?`, `SellAction` (enum: `FullSell` | `PartialSell` | `Keep`), `SellAmount?` (only used when `SellAction == PartialSell`).
- CSV schemas: see [doc/SLifeIrpRebalancer_상세_설계서.md](doc/SLifeIrpRebalancer_상세_설계서.md) §3.1 (principal-guaranteed) and §3.2 (funds). The fund CSV's return-period columns are dynamic — derive them from whichever periods the source HTML actually contains rather than hard-coding the five periods.

## When implementing

- The recommended phase order is in §6 of the spec: scaffold the NavigationView shell first, then HTML→CSV parsing, then account/sell-decision UI with proper enable/disable bindings (the partial-sell amount field must only be enabled when `SellAction == PartialSell`), then AI integration, then PDF.
- The user works in Korean; UI strings, error messages, and PDF content should be Korean. Code identifiers stay English.
- API keys go in Settings and should be persisted locally — do not hardcode or commit them.
